## 架构概览

Multica 自部署包含三个核心组件：

| 组件 | 说明 | 技术 |
|---|---|---|
| 后端 | REST API + WebSocket server | Go 单二进制 |
| 前端 | Web 应用 | Next.js |
| 数据库 | 主数据存储 | PostgreSQL 17 + pgvector |

如果用户希望在本机运行 AI agent，还需要安装 `multica` CLI，并在自己的机器上启动 agent daemon。

## 前置要求

- Docker 与 Docker Compose

## 快速安装

两条命令完成本地自部署和 daemon 配置：

```bash
curl -fsSL https://raw.githubusercontent.com/multica-ai/multica/main/scripts/install.sh | bash -s -- --with-server
multica setup self-host
```

这会安装 CLI、拉取 self-host 资源、从 GHCR 拉取官方镜像，并按 localhost 配好服务。完成后打开 http://localhost:3000。

登录方式有两种：

- 推荐生产方式：在 `.env` 配置 `RESEND_API_KEY`，重启 backend，验证码会发到邮箱。
- 本地测试方式：不配 Resend，验证码会打印到 backend 日志。

<Callout>
如果 self-host server 已经运行，只需要在 macOS / Linux 机器上安装 CLI，可以直接用 `brew install multica-ai/tap/multica`。
</Callout>

## 分步设置

### 1. 启动 server

```bash
git clone https://github.com/multica-ai/multica.git
cd multica
just selfhost
```

`just selfhost` 会自动创建 `.env`、生成随机 `JWT_SECRET`，并通过 Docker Compose 启动所有服务。

默认会拉取 GHCR 上最新稳定镜像。如果想从当前 checkout 构建 backend / web，运行：

```bash
just selfhost-build
```

启动完成后：

- 前端：http://localhost:3000
- 后端 API：http://localhost:8080

也可以手动执行 Docker Compose 流程：

```bash
cp .env.example .env
# 编辑 .env，至少修改 JWT_SECRET
docker compose -f docker-compose.selfhost.yml pull
docker compose -f docker-compose.selfhost.yml up -d
```

### 2. 登录

打开 http://localhost:3000。Docker self-host stack 默认设置 `APP_ENV=production`，且默认没有固定验证码。

- **推荐生产配置：** 设置 `RESEND_API_KEY`，重启 backend，真实验证码会发到邮箱。
- **未配置邮件：** 验证码会生成在服务端，并打印到 backend 容器日志，查找 `[DEV] Verification code for ...:`。
- **固定本地测试码：** 设置 `APP_ENV=development` 和 `MULTICA_DEV_VERIFICATION_CODE=888888`，重启 backend。`APP_ENV=production` 时固定码会被忽略。

`ALLOW_SIGNUP` 和 `GOOGLE_CLIENT_ID` 也在重启 backend / compose stack 后生效。web UI 会运行时从 `/api/config` 读取它们，不需要重新构建 web。

<Callout type="warning">
不要在公网可访问实例上设置 `MULTICA_DEV_VERIFICATION_CODE`。任何知道邮箱地址的人都可能用固定码登录。
</Callout>

### 3. 安装 CLI 并启动 daemon

daemon 跑在用户本机，不在 Docker 容器里。它会检测已安装的 AI agent CLI，向 server 注册运行时，并执行分配给智能体的 task。

安装 CLI：

```bash
brew install multica-ai/tap/multica
```

还需要至少安装一个 agent CLI，例如：

- Claude Code（`claude` 在 `PATH` 上）
- Codex（`codex` 在 `PATH` 上）
- Gemini CLI（`gemini` 在 `PATH` 上）
- OpenCode（`opencode` 在 `PATH` 上）
- OpenClaw（`openclaw` 在 `PATH` 上）
- Hermes（`hermes` 在 `PATH` 上）

一键配置：

```bash
multica setup self-host
```

它会自动：

1. 配置 CLI 连接 localhost
2. 打开浏览器完成登录
3. 发现你的 workspace
4. 在后台启动 daemon

自有域名部署时：

```bash
multica setup self-host --server-url https://api.example.com --app-url https://app.example.com
```

检查 daemon：

```bash
multica daemon status
```

也可以手动分步配置：

```bash
multica config set server_url http://localhost:8080
multica config set app_url http://localhost:3000
multica login
multica daemon start
```

### 4. 验证

1. 打开 http://localhost:3000 进入 workspace
2. 进入 **设置 → 运行时**，确认能看到你的机器
3. 进入 **设置 → 智能体** 创建新智能体
4. 创建 issue，并分配给该智能体

## 停止服务

```bash
just selfhost-stop
multica daemon stop
```

## 切回 Multica Cloud

如果之前在 self-host，现在想把 CLI 切回 [Multica Cloud](https://multica.ai)：

```bash
multica setup
```

它会重新配置 CLI、重新登录，并重启 daemon。已有 self-host 配置被覆盖前会提示确认。

<Callout>
本地 Docker 服务不会被自动停止。如果不再需要，需要单独停止。
</Callout>

## 升级

```bash
docker compose -f docker-compose.selfhost.yml pull
docker compose -f docker-compose.selfhost.yml up -d
```

如果想固定版本，在 `.env` 里把 `MULTICA_IMAGE_TAG` 设成精确 release，例如 `v0.2.4`。backend 启动时会自动运行迁移。

如果选中的 GHCR tag 尚未发布，可以回退到本地构建：

```bash
just selfhost-build
# 或
docker compose -f docker-compose.selfhost.yml -f docker-compose.selfhost.build.yml up -d --build
```

## 配置

全部配置都通过环境变量完成。建议从 `.env.example` 开始。完整清单见 [环境变量](/environment-variables)。

### 必填变量

| 变量 | 说明 | 示例 |
|---|---|---|
| `DATABASE_URL` | PostgreSQL 连接串 | `postgres://multica:multica@localhost:5432/multica?sslmode=disable` |
| `JWT_SECRET` | JWT 签名密钥；必须从默认值改掉 | `openssl rand -hex 32` |
| `FRONTEND_ORIGIN` | 前端访问地址，用于 CORS、cookie 与邀请链接 | `https://app.example.com` |

### 邮件

Multica 使用 Resend 发送验证码和邀请邮件。

| 变量 | 说明 |
|---|---|
| `RESEND_API_KEY` | Resend API key |
| `RESEND_FROM_EMAIL` | 发件地址，默认 `noreply@multica.ai` |

### Google OAuth

| 变量 | 说明 |
|---|---|
| `GOOGLE_CLIENT_ID` | Google OAuth client ID |
| `GOOGLE_CLIENT_SECRET` | Google OAuth client secret |
| `GOOGLE_REDIRECT_URI` | OAuth 回调地址，例如 `https://app.example.com/auth/callback` |

### 注册控制

| 变量 | 说明 |
|---|---|
| `ALLOW_SIGNUP` | 设为 `false` 可禁止新用户注册 |
| `ALLOWED_EMAIL_DOMAINS` | 允许注册的邮箱域名，逗号分隔 |
| `ALLOWED_EMAILS` | 允许注册的完整邮箱地址，逗号分隔 |

### 文件存储

评论图片和附件可以配置 S3 / S3 兼容存储，也可以使用本地磁盘。

| 变量 | 说明 |
|---|---|
| `S3_BUCKET` | 只填 bucket 名称，不要带 `.s3.<region>.amazonaws.com` 后缀 |
| `S3_REGION` | AWS 区域，默认 `us-west-2` |
| `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` | 静态凭证；不设时走 AWS SDK 默认凭证链 |
| `AWS_ENDPOINT_URL` | S3 兼容端点，例如 MinIO、R2、B2 |
| `CLOUDFRONT_DOMAIN` | CloudFront 域名；设置后公开 URL 使用该域名 |
| `CLOUDFRONT_KEY_PAIR_ID` | CloudFront signed cookie key pair ID |
| `CLOUDFRONT_PRIVATE_KEY` | CloudFront 私钥 PEM |

### Cookie

| 变量 | 说明 |
|---|---|
| `COOKIE_DOMAIN` | session 与 CloudFront cookie 的可选 `Domain` 属性 |

单主机部署（localhost、局域网 IP、单 hostname）请留空。只有当前端和后端在同一注册域的不同子域时才设置，例如 `.example.com`。不要设置为 IP 地址。

session cookie 的 `Secure` flag 会从 `FRONTEND_ORIGIN` 的协议自动推导：HTTPS 使用 secure cookie，HTTP 内网部署使用非 secure cookie，确保浏览器能存下登录态。

## 数据库

Multica 需要 PostgreSQL 17 和 pgvector 扩展。

使用内置 Docker Compose：

```bash
docker compose up -d postgres
```

使用自有 PostgreSQL 时确保扩展存在：

```sql
CREATE EXTENSION IF NOT EXISTS vector;
```

运行迁移：

```bash
./server/bin/migrate up
# 或源码方式
cd server && go run ./cmd/migrate up
```

## 手动运行（不使用 Docker Compose）

前置要求：Go 1.26+、Node.js 20+、pnpm 10.28+、PostgreSQL 17 + pgvector。

```bash
just build
DATABASE_URL="your-database-url" ./server/bin/migrate up
DATABASE_URL="your-database-url" PORT=8080 JWT_SECRET="your-secret" ./server/bin/server
```

前端：

```bash
pnpm install
pnpm build
cd apps/web
REMOTE_API_URL=http://localhost:8080 pnpm start
```

## 反向代理

生产环境建议在 backend 和 frontend 前放反向代理，负责 TLS 和路由。

### Caddy（推荐）

单域名部署：

```caddy
multica.example.com {
    @multica_ws path /ws /ws/*
    handle @multica_ws {
        reverse_proxy localhost:8080 {
            flush_interval -1
        }
    }

    reverse_proxy localhost:3000
}
```

前后端分域名：

```caddy
app.example.com {
    reverse_proxy localhost:3000
}

api.example.com {
    @multica_ws path /ws /ws/*
    handle @multica_ws {
        reverse_proxy localhost:8080 {
            flush_interval -1
        }
    }

    reverse_proxy localhost:8080
}
```

`/ws` 配置里有两个容易踩坑的点：

- `path /ws /ws/*` 要同时覆盖精确 `/ws` 和子路径，不要写成过宽的 `/ws*`。
- `flush_interval -1` 会关闭响应缓冲，避免 WebSocket 帧延迟。

### Nginx

```nginx
server {
    listen 443 ssl;
    server_name api.example.com;

    ssl_certificate     /path/to/cert.pem;
    ssl_certificate_key /path/to/key.pem;

    location / {
        proxy_pass http://localhost:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location /ws {
        proxy_pass http://localhost:8080;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_read_timeout 86400;
    }
}
```

前后端分域名时，设置：

```bash
FRONTEND_ORIGIN=https://app.example.com
CORS_ALLOWED_ORIGINS=https://app.example.com
REMOTE_API_URL=https://api.example.com
NEXT_PUBLIC_API_URL=https://api.example.com
NEXT_PUBLIC_WS_URL=wss://api.example.com/ws
```

## 健康检查

```text
GET /health
→ {"status":"ok"}

GET /readyz
→ {"status":"ok","checks":{"db":"ok","migrations":"ok"}}

GET /healthz
→ 与 /readyz 相同
```

`/health` 适合基础存活检查；`/readyz` 适合依赖感知的 readiness probe；`/healthz` 保留为运维习惯别名。
